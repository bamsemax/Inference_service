## Часть 3.
Возможные кандидаты (подумайте, какие из них важны именно для inference service и почему):
В целом кажется что все метрики полезны

### Latency P50 / P95 / P99 Показывает полное время ответа сервиса.
* P50 - Стандартное время ответа, с которым сталкиваются пользователи.
* P95 - Показывает время ожидания с которым сталкиваются большинство пользователей.
* P99 - Скорее содержит информацию о выбросах и хвостах.
Позволит заметить деградацию производдительности.
#### Целевые значения:
P99 < 300ms
P95 < 250ms
Для большинства пользователей такие значения не создадут дискомфорта.
При превышении порога:
* Найти причину, вызывающую задержку и устранить :)
* Например ограничить длину принимаемого текста если дело в модели


### Throughput (RPS)
Позволяет оценить пропускную способность сервиса на данной машине.
#### Целевые значения:
* > 10 RPS - разумное значение для CPU
  
При падении RPS меры могут быть схожими с таковыми для latency
Внедрить асинхронность
### Model inference time.
Может помочь в определении причины увеличения времени ответа.
С точки зрения пользовательского опыта latency важнее.
#### Целевое значение:
<100ms исходя из сопоставления с latency
При превышении ограничить длину текстов
Использовать батчинг если хватает памяти
### CPU / Memory usage. 
Позволит оценить насколько конкретная машиная справляется с задачей и оценить возможность выдержать рост нагрузки.
Также может объяснить рост latency.
В данной работе менее информативна, потому что метрика измерялась в условиях конкуренции с массой других приложений.

#### Приемлемые значения
Пиковые значения: CPU, RAM<80%
Вероятно сильно зависит от машины и режима её использования.
Не должно быть роста занятой RAM

При превышении: выделить больше ресурсов, проверить утечки памяти
